"Running shoes" translated word-for-word into German is "rennende Schuhe." No German types that into a search box. The phrase they actually search is "Laufschuhe." That gap, between the translation that is technically correct and the one that has real demand, is the whole problem this tool exists to solve, and it is why I do not trust the translator's first answer.

Multilingual SEO Tool answers one question before any money gets spent: of all the pages you could translate, and all the languages you could translate them into, which ones are worth a translator's time? Translating a page into five languages costs real money. The hard part is choosing the five. You give the tool a URL, it finds the page's keywords, translates them per locale, checks live search demand on each country's engine, and ranks what is worth translating first. It is in development, and what follows is how the design works, not a record of adoption it does not have.

Don't translate the words. Translate the intent, then let demand decide. The naive build asks a model for one translation per keyword and ships it. I did not do that. The translator proposes several intent-preserving variants for each locale instead of a single literal guess. Then a separate, non-negotiable step looks up the real monthly search volume for each candidate, and the winner is the variant that keeps the original meaning and has real demand on that country's search engine. The language model is allowed an opinion about meaning. It is not allowed the final say about what people search. That belongs to the volume data. This also catches the cases a single translation would silently miss: Spanish shoppers in Spain search "zapatillas," Mexican shoppers search "tenis," and a tool that returns one Spanish word for both is wrong for one of them.

Six steps, one stream. A LangGraph agent runs the page through a linear pipeline: URL, fetch, extract, translate, volume, score, stream. The split that matters is which steps are code and which are judgment. Fetching the page and scoring the candidates are pure deterministic code, the parts that should never improvise. Extraction and translation are the two steps that genuinely need a model, so those are the only two that get one. Every step streams its result to the UI the moment it finishes, so each locale's ranking appears as its score lands instead of after one long blocking wait.

The right model for each job. Pulling keywords out of a page is fast, high-volume, low-judgment work, so that runs on Groq's Llama 4 Scout. Translating a keyword while keeping its search intent intact is where quality decides the output, so that runs on Claude Sonnet 4.6. The translation prompts are cached, so the large shared parts of each request, the instructions that do not change between keywords, are not paid for twice. One model where speed wins, one where judgment wins, and a cache across the repeated boilerplate.

Validate before you pay, one cent at a time. The point of the tool is to spend a few cents confirming demand before spending a translator's hours on a page nobody searches for, so the demand check itself has to be cheap. Search volume comes from DataForSEO. A locale that produced dozens of candidate keywords does not fire dozens of calls. Every locale resolves in a single batched request of up to 1,000 keywords that bills one cent. The expensive thing the tool is trying to prevent, wasted translation, is gated behind the cheapest possible measurement.

Run it twice, pay once. Repeated runs are where naive pipelines quietly burn money. A SQLite checkpointer saves each run's final state to disk. Submit the same URL and the same locale set again and the saved result replays straight from disk, with no repeat model calls and no repeat search-volume calls. A 24-hour SERP cache sits underneath that, so repeated search-volume lookups inside that window do not re-bill the search provider either. Repeat work is free by construction, not by remembering to add a cache later.

What lands in the UI. Beyond the ranked list of which locales are worth the budget first, the front end scores every keyword variant on volume, competition, CPC, intent, and traffic value, with a short note on why a given variant won its locale. It surfaces hreflang gaps the page has not covered yet, and it flags the cross-dialect splits, the "zapatillas" against "tenis" cases, side by side so you can see exactly where one Spanish translation would have been wrong.

Own it, run it locally. There is no hosted service to sign up for. The tool is open source: clone the repo, drop in keys for the models and the search-volume provider, and run the whole pipeline on your own machine. The model is allowed to propose, the data decides, and you can read every step it took to get there. The repo is on GitHub at github.com/RealRogerWinter/multilingual-seo.

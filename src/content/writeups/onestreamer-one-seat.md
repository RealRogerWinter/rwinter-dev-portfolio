title: "One live seat is an architecture decision, not a gimmick"
description: "How a streaming platform that lets anyone seize the broadcast collapses into a single publisher, one SFU, and one perception pipeline that both fills the chat and bans abuse."
date: 2026-06-13
tags: ["webrtc", "agents", "onestreamer"]
---

OneStreamer started with one question: what if streamers competed for the audience by taking the broadcast away from each other? Only one person holds the stream at a time, and anyone watching can cut in and seize it, so the show keeps changing hands. That sounds like a game rule. It is really a constraint on the system, and the constraint is what makes the rest of the design fall into place.

Because the platform never runs more than one broadcast, the whole thing reduces to the simplest possible realtime shape: one publisher, fan-out to everyone. I run LiveKit as the only WebRTC SFU. A single publisher sends video in, and the SFU forwards it to every viewer at once. That one-to-many shape is exactly what the takeover game needs, and it is the shape a hundred parallel streams would never give me. When attention can only ever be on one feed, you do not pay the cost of routing many-to-many, and latency stays low enough that a takeover feels instant. A recorder taps the same feed continuously, so any moment can be clipped after the fact without a second capture path.

The seat itself is the game. Holding it is the whole thing, and it is always up for grabs. Between rounds, streamers and viewers pick up items. Some raise a stream's defense so the seat cannot be grabbed. Others drain that defense or disrupt the broadcast, opening a window for the next takeover. So the loop is: interrupt, attack the defense, seize the stream. When a challenger cuts in, the current broadcast stops and the challenger becomes the show. None of that needs new plumbing, because there is still only ever one live publisher. The takeover is a swap of who occupies the single seat, not a new stream coming online.

The second idea also falls out of "one feed." The room is not empty between humans, and it is not only humans when someone is live. OneStreamer seeds the chat with LLM-powered viewers. Each AI viewer takes screenshots of the stream and a transcript of the audio, feeds both into a lightweight model, and posts back to chat. Because the reactions come from the current frame and the current audio, they track what is on screen right now instead of running a generic script. Every viewer keeps a consistent personality across a session, so running jokes and callbacks build over time. The grounding is the point: frames plus audio transcript in, chat out, all of it anchored to the live moment rather than a canned loop.

Here is the part that justifies building it this way. Moderation runs on the same pipeline as the viewers. Not a separate model with its own capture: the same eyes and ears. Frames from the live video and the audio transcript pass through a detection step, and when something crosses the line the offending account is banned automatically, without waiting for a human to catch it. I reuse the exact perception path that already makes the AI viewers react. If a viewer agent can watch the stream well enough to make a joke about what just happened on screen, the same inputs are good enough to catch what should not be on screen. One perception pipeline, two jobs: populate the room, and keep it safe.

Viewers are not stuck behind the glass either. They can change what the broadcast looks and sounds like in real time: draw on the live video with graphics layered over the feed, trigger sound effects that play through the broadcast audio for everyone, and speak into the stream with text to speech so chat can talk out loud mid-broadcast. Because everyone is watching the one feed, an effect a viewer triggers lands for the whole audience at once. That is the same one-to-many property paying off again, this time for input instead of output.

The stack is local-first where it can be. STT runs on whisper.cpp, the default LLM is a local mistral on Ollama, with Groq available as an optional cloud model. The realtime media is LiveKit as the sole SFU, noted in ADR-0024, with coturn for TURN/STUN and ffmpeg, streamlink, and yt-dlp handling URL-relay ingest. The app is React 19 and TypeScript on a Node 18+, Express, and Socket.IO backend; SQLite for data, Backblaze B2 for recordings and clips, Passport with Google OAuth and Turnstile for auth, Strapi for the blog, and PM2 behind nginx in production.

OneStreamer is also where I test how far AI agents can carry a real product, and its own history is a record of that. The first prototype was hand-assembled by copy-pasting code out of an early ChatGPT, one piece at a time, before agents could touch a repo. Then Claude Code took over the codebase on Sonnet 4.5, and whole features started landing in a sitting instead of a weekend. Switching to Opus made larger, system-wide edits hold together, refactors that touched many files at once. The current version is a deep refactor done with Opus 4.8, and it keeps going from here.

The lesson I keep relearning is that a sharp constraint pays you back. "Only one person holds the stream" reads like a rule for players. In the code it is the reason the SFU is one-to-many, the reason a single recorder can clip anything, the reason one perception pipeline can both fill the chat and police it, and the reason a viewer's effect reaches the whole room. The game and the architecture are the same decision.

Price Games is a free browser game about one tricky skill: knowing what things actually cost. Seven modes, real Amazon prices, live multiplayer, no signup. You open a tab and play. Pricey, my Twitch streamer, plays it too, on the same engine humans use. That part you can see. The part you can't see is who wrote it, and that's the part I find most interesting: Price Games was authored end to end by Claude Code agents, mostly Opus 4.7 and 4.8, and the only thing a human does in the loop is approve the merge.

I want to be precise about what "end to end" means, because it's easy to say and easy to fake.

Every change starts with a failing test. Not the code, the test. The agent writes the test that describes the behavior it's about to add, watches it fail for the right reason, and only then writes the code to make it pass. Test-driven development is the contract, and it does more work here than it does in a human's hands. A failing-then-passing test is the cheapest possible proof that the agent did the thing it claimed, on a real CI runner, not in a confident paragraph. When the author can't be trusted to know whether it succeeded, you make the build tell you. That's the whole posture.

The agent opens its own PR. Then it spawns sub-agents: one for code review, one for security review. Those reviewers read the diff fresh and file findings, and the original agent implements them, the same way it would act on a human reviewer's comments. The split matters. The agent that wrote the change is the worst judge of it, so review goes to a different context with a different job. After the fixes land, the agent watches CI through to green over MCP, polling the pipeline status itself instead of waiting to be told. Only when it's green does a human look. The merge button is the last thing a human touches, and by the time I reach it, the failing test, the review, the security pass, and the CI run already exist as a paper trail in the PR.

CircleCI does the rest: build, tests, push the Docker image, deploy. The agents document the work in PR comments and in the project docs as they go, so the record of why a thing was built sits next to the thing. This is the loop I keep tightening, and Price Games is where I push how much of it agents can own.

The reason I trust this on something real is that the something is real. People play this. Which means the surface the agents have to get right is not a toy. The frontend is React 18 on Vite 7 with TypeScript and React Router 7. The backend is Express and Socket.IO over SQLite through better-sqlite3, with the realtime layer carrying live multiplayer rooms. Auth is its own minefield the agents had to clear: OAuth 2.0 against Google, Facebook, and Amazon, bcryptjs, httpOnly sessions, Cloudflare Turnstile, and TOTP for admins. Email runs on Resend, push is self-hosted Web Push over VAPID, infra is Docker and docker-compose behind Caddy for automatic HTTPS, with Tailscale fencing off admin access. None of that is the kind of thing you hand-wave through. It either works when a stranger logs in from a phone, or it doesn't.

The game data is built to be structured, not scraped-and-shrugged. Every product and price comes from an actual Amazon listing. An AI pipeline scrapes those listings and classifies each one by category and price tier, and the game engine uses those labels to build matchups worth playing. In Comparison mode that's the difference between two genuinely comparable items and a random pair where the answer is obvious. The classification isn't decoration. It's what keeps the core question hard.

Price Games has a second job, and it's the reason it earns its keep as an engineering project: it's a test bed for growth instrumentation I'd rather build than rent. UTM tracking is a custom tagging system that attributes every visit to its source campaign. Short links and QR codes carry that same UTM data from print and screens into one pipeline. A purpose-built analytics layer records games played and how a session moves between modes, and where each game started: homepage, room link, quickplay, or the game browser. Self-hosted Web Push brings players back for giveaways and new modes. A running leaderboard and a monthly giveaway give regulars something to chase. Building these myself means I see exactly how attribution works, and I can instrument the questions I care about instead of the ones a vendor decided to answer.

The honest summary is that this is the project where I find out how far the agent loop reaches. Not a code-completion demo, not a weekend script. A live multiplayer web app with OAuth, the session and admin-2FA hygiene real logins demand, realtime rooms, an analytics stack, and a deploy pipeline, written test-first by agents that review their own work and prove it on CI before I ever read the diff. The legibility is the point. I can watch every change happen as a failing test, a review thread, and a green build, and the merge is the one decision I keep for myself.

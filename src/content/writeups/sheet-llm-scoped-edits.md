---
title: "The edit that refuses to touch what you didn't ask it to"
description: "How sheet-llm turns \"rewrite bars 5 through 8\" into a server-side promise that bars 1 through 4 do not move."
date: 2026-06-14
tags: ["llm", "architecture", "sheet-llm"]
---

Ask sheet-llm to rewrite bars 5 through 8 of a waltz, and bars 1 through 4 come back byte for byte identical. Not "usually." Not "the model tried." The server hashes the bars that were supposed to stay, and if any of them moved, it refuses the edit. That refusal is the whole design. Everything else, the staff rendering, the model routing, the chat that explains a ii-V-i, hangs off that one guarantee.

sheet-llm is a notation editor you talk to. You describe music in plain language, and a language model writes it onto the staff. It is in development, so this is a writeup of how it works and why it is built the way it is, not a victory lap. The interesting decision is not that a model can compose. It is that the model is held to a contract: change only what was asked, and prove it.

**Treat every edit as additive.** Most AI editing tools regenerate the whole document and hope the parts you liked survive. sheet-llm does the opposite. New bars get added. Named bars get rewritten. Everything else is left strictly alone. The enforcement lives on the server, not in the prompt: before a change reaches the saved score, one gate hashes the bars meant to be preserved and rejects the edit if a single one changed. A second gate watches for wholesale rewrites and holds them behind an explicit confirmation. A full rewrite can still happen. It just never happens by surprise.

That is a stronger promise than "the model is careful," because it does not depend on the model being careful. It depends on a hash.

**The Score is the spine, and that is what makes the hash possible.** This only works because the model is not emitting notation markup. It emits a structured Score, written as validated JSON and checked with Zod. The staff a musician sees is rendered from that Score (ABC, then abcjs to SVG). So is the playback. So are the MusicXML, MIDI, and PDF exports. The data is the single source of truth, and the notation is always derived from it, never the other way around. When the source of truth is a typed document with addressable bars, "did bars 1 through 4 move" becomes a literal equality check on a subtree. Ask a model for SVG or engraving markup and that question has no clean answer. Ask it for a Score and the answer is a hash comparison.

**One orchestrator, several models, picked per request.** Every chat turn runs through a single orchestrator. It reads the prompt and the current score, clears a copyright filter, then reasons about which edit to make. Its first real job is to read intent and classify the request by complexity. "Make bar 2 staccato" is a small change, so it goes to a smaller, faster model. "Harmonize this melody as a chorale" needs more, so it goes to a more capable one. And the same classifier notices when a request is not an edit at all: "why does bar 2 feel unresolved" gets routed to a conversational model that answers in words, with the open score as the reference point. The work is matched to the cost and the capability every time, instead of paying frontier prices to set one articulation.

Because questions get their own path, the editor doubles as something to talk to. You can ask what key a piece is in, or what a ii-V-i is, and it can teach by doing: it writes the example straight onto the staff, so the explanation of a scale or a cadence arrives with notes you can see and play.

**What the musician sees is a staged proposal, not a fait accompli.** Whatever survives the gates is rendered as a ghost preview. The new bars glow amber on the staff while the saved score sits untouched underneath. You accept or reject. Only on accept does anything land, and every turn is saved as a versioned checkpoint you can replay. The loop is short: describe, refine in chat, review the amber edit, and reach in by hand whenever a note needs moving, because the assistant is one more tool on the canvas, not a wall between you and the page.

**Build it in the open, on purpose.** sheet-llm is open source and runs locally, and the model provider is swappable: a local model on the same machine, your own API key for any provider, or a hosted model. The codebase abstracts over providers (the Anthropic SDK is the default behind that seam), so nothing about the editor is welded to one vendor. The repo, the CircleCI pipeline, and the runtime image on GHCR are all public, on the principle that a control which only works because it is hidden is not a control. Secrets are excluded by construction, never living in the repo, image, or logs, and every deploy still waits on an explicit human approval. The pipeline reads the same way: a verify gate, then an image push, then a manual-approval deploy.

The rest of the stack follows the same plainness. Next.js 16 and React 19 on the App Router, TypeScript 5. SQLite through better-sqlite3 and Drizzle for persistence, Zustand for client state. Sessions are anonymous by default with optional accounts, signed with jose JWTs over argon2id and OAuth via arctic. Import comes back in through fast-xml-parser; export goes out as MusicXML, MIDI, and PDF (jspdf with svg2pdf). It is packaged with Docker and docker compose, fronted by Caddy for automatic HTTPS, with Cloudflare in front and Litestream streaming the database to R2, on a self-hosted VPS behind origin mTLS.

None of that is the headline. The headline is the refusal. A language model is a wonderful way to turn a sentence into music and a dangerous way to edit a document you care about, because its instinct is to rewrite everything it touches. sheet-llm keeps the first behavior and fences off the second with a structured Score and a hash that says no. You can watch exactly what an edit changed, because the system can prove exactly what it did not.

Most recommenders never find out if they were right. They suggest ten titles, you click one, and the loop ends there. This skill closes it from the other side. It is a `/recommendations` Claude Code skill for Plex that scans your library, researches the web for titles you do not own but will love, and publishes a static site of cards plus an interactive constellation map. The part I care about is what happens on run two: when a title it recommended last time shows up in your library this time, it marks that pick `acquired`, with the date, and folds it back into your taste profile as a strong "yes, more of this." There is no labeled training set anywhere in the project. The next library scan is the eval.

That reframing is the whole design. A recommendation is a hypothesis. Acquiring it is the graded answer. The bookkeeper that scores it is `track_state.py`, a standard-library Python script with no pip dependencies, and its job is dull on purpose: load this run's snapshot, load the previous one, diff them by a normalized `title+year+type` key. Titles you added that match a standing recommendation get promoted to `acquired`. Titles you added that nobody suggested are logged as taste the system under-served. Standing picks you keep skipping are not deleted; they are kept and gently down-weighted. Recommendations accumulate across rounds and never reset, so the ledger is a running record of guesses and outcomes rather than a fresh slate each time.

Because the evaluation is real, the learning can be too. The diff feeds prompt `07-evolve-preferences`, which rewrites `preferences.json` with `leanInto`, `easeOffOf`, and `nextRoundGuidance` fields. The next research pass reads that, plus an exclusion set of everything you already own and every pick still standing from prior rounds, so each round is forced onto new ground instead of re-suggesting the same safe titles. The site surfaces this with a "Since last time" panel, so the owner can watch the system react to what they did. Build in public extends to the model's own behavior here: the feedback is legible because it is grounded in a fact you can check, your shelf, not an opinion the model holds about you.

The second decision is what makes that grounding possible: every recommendation must tie to a specific title you own, with a specific reason. Not "both are sci-fi." The research prompt demands one to three named owned anchors per pick with a concrete relationship and a strength score, and the rule is enforced twice. The prompts refuse a pick with no honest "because you have X," and `build_site.py` dedupes again as a safety net on the same normalized key. A pick with no bridge to the library does not ship. This is what lets the loop work at all: a recommendation that is anchored to *Annihilation* is falsifiable, because either you go acquire something it pointed you to or you do not, and the anchor is what the next diff can score against.

Those anchors are also the geometry. The map is not a decorative graph. Prompt `05-compute-proximity` approximates a 2-D embedding by hand: it clusters every node into four to nine taste regions, then places each recommendation *between* its strongest owned anchors and the cluster center, emitting normalized `x,y` coordinates and one edge per `relatedTo` link weighted by strength. The frontend runs a light force simulation to de-overlap, so the prompt only has to be well-clustered, not pixel-perfect. Proximity encodes similarity and every edge is a "because you have X" line you can trace. The recommendation logic and the picture are the same data structure, which is why selecting a per-user tab can re-render the whole constellation to just that person's picks client-side from each rec's links, with no extra layout pass.

The engineering around the LLM stages is deliberately boring so the interesting stages stay legible. The pipeline is `prompts/01` through `08`, each a small prompt with a strict JSON contract, orchestrated by a Workflow script that fans the slow part, web research, out in parallel, one subagent per taste region. Research triangulates across at least two sources, TMDB and Trakt and TasteDive and Letterboxd and Reddit among them, treating ratings as a filter and not a goal, because a 6.8 cult title can beat a 7.9 crowd-pleaser for the right shelf. Deterministic Python handles scanning, diffing, assembly, and deploy. Nothing personal is committed: the Plex token is read at runtime from `Preferences.xml`, falling back to `docker exec` or `sudo` when the file is mode-600, and it is never stored or printed. The published site is pure static HTML, CSS, vanilla JS, and D3 from a CDN, with no build step and no Node assumed on the host.

There is even a measurement tool for the bold runs. Two optional modes, daring and discovery, push the research past the safe zone or synthesize a brand-new taste cluster from your genre trends. `analyze_runs.py` then tables every target by mode and reports *anchor novelty*: how much a daring run reused the standard run's library anchors versus reaching for new shelves. The README is honest that on a thin basis, under roughly 20 distinct titles, daring drifts newer instead of deeper and discovery leans on generic prestige priors. The point of measuring is to tell whether a bold run broke new ground, and which prompt to tune when it did not. That is the same instinct as the core loop: do not trust the system because it sounds confident, check it against something real.
